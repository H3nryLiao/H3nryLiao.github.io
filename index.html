Leetcode0026 删除有序数组中的重复项

Posted on 2022-05-20 In Leetcode

题目描述

给你一个 升序排列 的数组 nums ，请你 原地 删除重复出现的元素，使每个元素 只出现一次 ，返回删除后数组的新长度。元素的 相对顺序 应该保持 一致 。

由于在某些语言中不能改变数组的长度，所以必须将结果放在数组nums的第一部分。更规范地说，如果在删除重复项之后有 k 个元素，那么 nums 的前 k 个元素应该保存最终结果。

将最终结果插入 nums 的前 k 个位置后返回 k 。

不要使用额外的空间，你必须在 原地 修改输入数组 并在使用 O(1) 额外空间的条件下完成。

题解

本题解法简单，一次遍历整个数组，使用两个指针i、ptr和一个暂存元素cur即可。i指向当前所遍历的元素，ptr指向当前元素应插入的位置，暂存元素保存当前想要查询的相同数字。cur初始化为数组第一个元素，遍历过程中，若当前元素等于cur，说明是重复元素，数组长度减一，若当前元素不等于cur，说明是下一不同数字的第一个数组元素，根据ptr将元素插入数组并更新ptr，同时更新cur为当前数字。

从题解来看，做法是一致的，类似这样的两个指针：一个负责遍历，一个负责指示插入位置的做法叫做快慢指针。

Leetcode0027 移除元素

Posted on 2022-05-20 In Leetcode

题目描述

给你一个数组 nums 和一个值 val，你需要 原地 移除所有数值等于 val 的元素，并返回移除后数组的新长度。

不要使用额外的数组空间，你必须仅使用 O(1) 额外空间并 原地 修改输入数组。

元素的顺序可以改变。你不需要考虑数组中超出新长度后面的元素。

题解

本题关键点有两个，一是找出值为val的元素，二是移除元素后对剩余元素进行移动的操作

如果直接遍历查找val，对于每一个找到的目标元素，都要将其后的所有元素前移，时间复杂度很高

最开始想到的解法是，先对数组排序，排序后值为val的元素就都在一块了，对剩余元素的移动只用做一次即可

还没实现上面的解法前，看示例时发现

对于输入：nums = [0,1,2,2,3,0,4,2]，示例输出是：5, nums = [0,1,4,0,3]

结果数组中，元素4和0排到了3的前面，很难不让人联想到双指针的做法

基本思想是，头指针从前往后遍历，尾指针从后往前遍历，只要头指针遍历到等于val的元素，从尾指针开始往前找不等于val的第一个元素，插入到头指针所指位置，这样时间复杂度只有O(n)

实际在实现的过程中，发现了这样一些问题：数组长度为0和1的情况特殊，需要单独处理，并且在头尾指针移动过程中也有很多考虑的点，导致最后代码可读性很低。

题解中给出的双指针和自己想到的类似，但是有一点细微的不同：只要头指针遍历到等于val的元素，就将尾指针指向的元素插入头指针所指位置，然后尾指针左一，头指针只有在当前元素不等于val时才右移

Leetcode0015 threeSum

Posted on 2022-05-17 In Leetcode

题目描述

给你一个包含 n 个整数的数组 nums，判断 nums 中是否存在三个元素 a，b，c ，使得 a + b + c = 0 ？请你找出所有和为 0 且不重复的三元组。

注意：答案中不可以包含重复的三元组。

题解

自己写这道三数之和的题目时的思路历程：

  1. 采用三重循环进行遍历，使用一个flag数组标志元素是否被访问过以解决重复问题

    问题：正确的结果二维数组应该是一维数组元素内具有递增顺序，二维数组内的各个一维数组也按维度具有递增顺序

    而直接三重遍历得到的结果是遍历顺序而非递增

  2. 解决方法：对于得到的三个元素先进行组内排序，在将这组元素append到二维数组时进行组间排序

    问题：这样的额外操作及其耗时，是直接导致后面的超时原因之一

  3. 再次进行测试，出现了结果数据遗漏的情况，发现是采用flag数组的原因：

    flag数组的执行逻辑是只要当前的三个数字之和为0，即标记这三个数字为已访问，以后不再考虑

    对于**[3,0,-2,-1,1,2]**这样的测试用例，得到第一组解[-2, -1, 3]后，3、-2、-1在之后的循环都不再访问，但其实还有[ -2, 0, 2]、[-1, 0, 1]这样的解，说明不能简单的认为和为0的三个数不能再组成解

  4. 解决方法：去掉flag数组，在append时进行去重

    问题：解答正确，但超时

Leetcode0016 最接近的三数之和

Posted on 2022-05-17 In Leetcode

题目描述

给你一个长度为 n 的整数数组 nums 和 一个目标值 target。请你从 nums 中选出三个整数，使它们的和与 target 最接近。

返回这三个数的和。

假定每组输入只存在恰好一个解。

题解

这道题和0015很相似，思路还是先将数组按升序排列，遍历的过程中用双指针去更新离target最近的三数之和

在有序数组中找最近三数之和的思路就是：

根据遍历的第一个数，指针一指向的第二个数和target，由target - num1 - num2求出目标数num3，**使得和最接近的第三个数要么是数组中小于num3的第一个元素，要么是大于num3的第一个元素，要么等于num3**。指针二从队尾开始查找，对应这三种情形：1、当前元素大于num3，指针二左移，去找更接近的数；2、当前元素等于num3，直接返回target；3、当前元素小于num3，此时指针二指向的元素即为小于num3的第一个元素，比较较小元素和较大元素。

Leetcode0002 两数相加

Posted on 2022-05-17 In Leetcode

题目描述

给你两个非空的链表，表示两个非负的整数。它们每位数字都是按照逆序的方式存储的，并且每个节点只能存储一位数字。

请你将两个数相加，并以相同形式返回一个表示和的链表。

你可以假设除了数字 0 之外，这两个数都不会以 0 开头。

题解

题目已经明确提示了使用单链表，逆序存储也为我们解决了数字相加的对齐问题，降低了求解的难度

主要的注意点就在于：

  1. 进位问题
  2. 某一链表扫完而另一个链表有剩余的情况
  3. 最高位存在进位（结果最高位为1）

解决进位问题就使用一个 flag 标识位即可，当前 node 存在进位则置为 1 ，无进位则置为 0 ，求和时直接Val1 + Val2 + flag

解决第二个问题很简单，将剩对链表复制到结果中即可，但复制时依然要考虑进位

解决最高位进位问题，在两个链表均扫完后，检查 flag 是否为 1 ，为 1 则new一个节点存 1 即可

Leetcode0011 Container with Most Water

Posted on 2022-05-10 Edited on 2022-05-17 In Leetcode

题目描述

给定一个长度为 n 的整数数组 height 。有 n 条垂线，第 i 条线的两个端点是 (i, 0) 和 (i, height[i]) 。

找出其中的两条线，使得它们与 x 轴共同构成的容器可以容纳最多的水。

返回容器可以储存的最大水量。

说明：你不能倾斜容器。

题解

最开始想到的办法就没什么脑子，暴力去便利整个数组，找到最大水量，时间复杂度就是O(n^2)

但第一次提交后报超时，说明解法是不对的

思来想去也没找到解法，无奈看了题解

正确的做法是使用双指针，初始化为指向数组第一个元素和最后一个元素，其意义是可作为容器边界的所有位置的范围

每次迭代，都修改高度较小那一边的指针，遍历完整个数组即得到最大水量，时间复杂度为O(n)

但是为什么要修改高度较小的指针？为什么这样遍历一次就能得到最大水量？

首先理解双指针的含义，双指针包含的范围内，是可作为容器边界的所有位置，初始时head = 0 tail = len(height) - 1表示容器边界在整个数组下标范围内都有可能

此时双指针对应的面积宽度width = tail - head，水量area = min(height[head], height[tail]) * width

假设height[head] < height[tail]，此时保持head指针不变，向左移动tail指针，宽度width会减小，高度因为要取min(height[head], height[tail'])，所以new height只会小于等于old height，也就是说，双指针中高度较小者指向的元素是当前的短板，会限制水量的增加，如果要找更大的水量，这块“板子”肯定是不行了，要去找更高的板子，所以每次迭代要修改高度较小的指针。

合并分支时解决冲突

如果不同分支对同一文件作了不同修改，那么在合并这些分支时就会出现冲突——即Git无法确定哪个分支所作的修改是应该保留的

出现冲突后，Git会在文件中出现冲突的位置作标记，如：

要解决冲突只用修改冲突文件并重新提交即可

重新提交后，冲突的分支会被自动合并，不要忘记删除分支

可以使用命令git log --graph查看分支的合并情况

分支管理策略

在合并分支时，Git默认使用Fast forward模式，即将master分支直接指向其他分支的提交

在这种模式下，删除分支后会丢掉分支信息

如果禁用Fast forward模式，Git在合并时会保留分支信息

使用Fast forward模式，看不出来曾经做过合并，时间线上不会有ff模式合并的分支

有个疑问，no-ff模式合并后是否需要删除dev分支？

经过测试，删除dev分支后，时间线仍保留了分支合并时的节点以及版本号

在分支中修复Bug

假设这样一个场景：当前正在dev分支上工作，进行到一半必须紧急修复master分支上的某一个bug

应该这样做：

  1. 保存未做完的工作

    使用git stash命令暂存工作现场，这样使用git status查看工作区就没有需要提交的修改

  2. 修复master分支的Bug

    在master分支创建临时分支，比如创建一个fixbug分支，并修复Bug后合并

  3. 恢复dev分支未完成的工作

    使用命令git stash list查看暂存的数据列表，在对应分支下使用git stash pop恢复暂存的内容，同时将其从stash区中删除

    若stash list中有多条记录，需要恢复指定stash时，使用命令git stash pop stash@{<index>}

需要注意：一是必须在暂存记录对应的分支下恢复，否则恢复后无法切换分支

二是删除分支后，对应stash中的数据不会被删除，需要手动git stash drop stash@{<index>}

对于上面的场景，dev分支来自master分支，那么master分支中存在的bug也存在于dev分支中

如何快速的修复dev分支中的bug？使用git cherry-pick命令

将之前临时分支提交的版本号作为该命令的参数即可
